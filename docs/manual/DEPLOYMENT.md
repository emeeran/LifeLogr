## 1. Development Setup

### Prerequisites

- **Python 3.11+** with `uv` package manager
- **Node.js 20+** with npm
- **Rust** (for Tauri desktop builds)
- **Git**

### Quick Start

```bash
# Clone the repo
git clone https://github.com/your-org/dailybyte.git
cd dailybyte

# Install backend dependencies
cd backend
uv sync

# Install frontend dependencies
cd ../frontend
npm install

# Launch both servers (auto-picks available ports)
cd ..
chmod +x dev.sh
./dev.sh
```

### Manual Start

```bash
# Terminal 1: Backend
cd backend
uv run uvicorn app.main:app --port 8000 --reload

# Terminal 2: Frontend
cd frontend
npm run dev
```

The frontend dev server proxies API requests to the backend via Vite's proxy config.

## 2. Building for Production

### Frontend Build

```bash
cd frontend
npm run build
```

Output: `frontend/dist/` — static HTML/CSS/JS files ready to serve.

### Backend Build

The backend runs directly from source in production:

```bash
cd backend
APP_ENV=production uv run uvicorn app.main:app --host 0.0.0.0 --port 8000
```

For packaged desktop apps, the backend is bundled into a single binary via PyInstaller (see Section 5).

### Production Static File Serving

When `APP_ENV=production`, the FastAPI backend serves the built frontend from `frontend/dist/` automatically. Only the backend needs to be deployed.

## 3. Docker Deployment

### Dockerfile

The project includes a multi-stage Dockerfile:

```bash
# Build the image
docker build -t dailybyte .

# Run the container
docker run -d \
  -p 8000:8000 \
  -v dailybyte-data:/data \
  -v dailybyte-media:/app/media \
  --name dailybyte \
  dailybyte
```

### Docker Compose

```bash
# Start
docker compose up -d

# Stop
docker compose down

# View logs
docker compose logs -f

# Rebuild after changes
docker compose up -d --build
```

**docker-compose.yml configuration:**

```yaml
services:
  dailybyte:
    build: .
    ports:
      - "8000:8000"
    volumes:
      - dailybyte-data:/data
      - dailybyte-media:/app/media
    environment:
      - APP_ENV=production
      - SECRET_KEY=${SECRET_KEY}
    env_file:
      - .env
    restart: unless-stopped

volumes:
  dailybyte-data:
  dailybyte-media:
```

### Environment File

Create a `.env` file in the project root:

```env
SECRET_KEY=your-random-secret-key-here
APP_ENV=production
CORS_ORIGINS=http://your-domain.com
OLLAMA_BASE_URL=http://host.docker.internal:11434
```

## 4. systemd Service

For running as a system service on Linux:

### Service File

The service file is generated by `build-web-deb.sh` during packaging. For manual setup, create a service file:

```ini
[Unit]
Description=DailyByte Journal App
After=network.target

[Service]
Type=simple
User=dailybyte
WorkingDirectory=/opt/dailybyte
ExecStart=/opt/dailybyte/backend/.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000
Environment=APP_ENV=production
Environment=SECRET_KEY=your-secret-key
Environment=DATA_DIR=/var/lib/dailybyte
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

### Installation

```bash
# Copy service file (or use the one from build output)
sudo cp dailybyte.service /etc/systemd/system/

# Create data directory
sudo mkdir -p /var/lib/dailybyte/media
sudo chown dailybyte:dailybyte /var/lib/dailybyte

# Enable and start
sudo systemctl daemon-reload
sudo systemctl enable dailybyte
sudo systemctl start dailybyte

# Check status
sudo systemctl status dailybyte

# View logs
sudo journalctl -u dailybyte -f
```

### Nginx Reverse Proxy

```nginx
server {
    listen 80;
    server_name diary.example.com;

    client_max_body_size 25M;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Long timeout for AI operations
        proxy_read_timeout 300s;
    }
}
```

## 5. Desktop Packaging (Tauri)

### Architecture

```
┌──────────────────────────────────────┐
│  Tauri Native Shell (Rust)           │
│  ┌────────────────────────────────┐  │
│  │  WebView (Vue.js Frontend)     │  │
│  └────────────────────────────────┘  │
│  + Sidecar: PyInstaller Backend      │
│  + Single Instance Lock              │
│  + Platform Data Directory           │
└──────────────────────────────────────┘
```

### Prerequisites

- **Rust** stable toolchain: `rustup default stable`
- **Tauri CLI**: `cargo install tauri-cli`
- **PyInstaller**: `pip install pyinstaller`
- **System libraries** (Linux only):
  ```bash
  sudo apt install libwebkit2gtk-4.1-dev libgtk-3-dev libayatana-appindicator3-dev librsvg2-dev
  ```

### Building

#### Linux (AppImage + DEB)

```bash
cd desktop
make build-linux
```

Output:
- `desktop/src-tauri/target/release/bundle/appimage/` — AppImage
- `desktop/src-tauri/target/release/bundle/deb/` — Debian package

#### Windows (MSI)

```bash
cd desktop
make build-windows
```

Output: `desktop/src-tauri/target/release/bundle/msi/`

Requires Windows build environment or cross-compilation setup.

#### macOS (DMG)

```bash
cd desktop
make build-mac
```

Output: `desktop/src-tauri/target/release/bundle/dmg/`

Requires macOS build environment.

### Build Steps Explained

1. **Frontend build**: `npm run build` → `frontend/dist/`
2. **Backend binary**: PyInstaller bundles `backend/` into `dailybyte-backend`
3. **Sidecar placement**: Binary copied to `src-tauri/binaries/` with platform suffix
4. **Tauri build**: Rust compiles the native shell, bundles frontend + sidecar

### Data Directory

The Tauri shell passes `DIARI_DATA_DIR` to the backend sidecar:

| Platform | Path |
|----------|------|
| Linux | `~/.local/share/com.dailybyte.desktop/` |
| Windows | `%APPDATA%\dailybyte\` |
| macOS | `~/Library/Application Support/dailybyte/` |

### PyInstaller Spec

The backend is bundled using `desktop/scripts/pyinstaller.spec`:

- Includes all required Python packages
- Excludes heavy optional dependencies (Whisper, Tesseract, WeasyPrint)
- Produces a single executable with no Python installation required
- Stripped and UPX-compressed for smaller binary size

## 6. Mobile Packaging (Capacitor)

### Architecture

```
┌──────────────────────────────────────┐
│  Capacitor Native Shell              │
│  ┌────────────────────────────────┐  │
│  │  WebView (Vue.js Frontend)     │  │
│  └────────────────────────────────┘  │
│  + @capacitor-community/sqlite       │
│  + Direct SQLite (no Python)         │
│  + Feature-flagged desktop features  │
└──────────────────────────────────────┘
```

Mobile uses **direct SQLite** via `@capacitor-community/sqlite` — no Python runtime is needed. The `MobileProvider` replicates all CRUD operations in TypeScript.

### Prerequisites

- **Node.js 20+** with npm
- **Android Studio** (for Android builds)
- **Xcode 15+** (for iOS builds, macOS only)
- **Java 17** (for Android builds)

### Android Build

```bash
cd mobile

# First time setup
make init

# Build and open in Android Studio
make android

# Or build APK directly
make build-android
```

Output: `mobile/android/app/build/outputs/apk/release/`

### iOS Build

```bash
cd mobile

# First time setup
make init

# Build and open in Xcode
make ios

# Or build from command line
make build-ios
```

### Feature Flags on Mobile

Desktop-only features are automatically disabled:

| Feature | Desktop | Mobile |
|---------|---------|--------|
| OCR (Tesseract) | ✅ | ❌ |
| Whisper Transcription | ✅ | ❌ |
| PDF Export (WeasyPrint) | ✅ | ❌ |
| TTS (edge-tts) | ✅ | ❌ |
| AI (Ollama) | ✅ | ❌ |
| Cloud Sync | ✅ | ❌ |
| Templates | ✅ | ✅ |
| Search (FTS5) | ✅ | ✅ |
| Media Upload | ✅ | ✅ |
| Geotagging | ✅ | ✅ |
| Reminders | ✅ | ✅ |
| Analytics | ✅ | ✅ |

The frontend checks `import.meta.env.VITE_PLATFORM === 'capacitor'` to hide desktop-only UI elements.

### Building the Frontend for Mobile

```bash
cd frontend
VITE_PLATFORM=capacitor npm run build
```

This sets the `VITE_PLATFORM` env var which enables the mobile code path.

## 7. CI/CD Pipeline

### GitHub Actions

The project includes `.github/workflows/build.yml` for automated multi-platform builds.

**Triggers:**
- Push a version tag: `git tag v1.0.0 && git push --tags`
- Manual dispatch via GitHub Actions UI

**Jobs:**

| Job | Runner | Output |
|-----|--------|--------|
| `build-linux` | ubuntu-latest | AppImage + .deb |
| `build-windows` | windows-latest | .msi |
| `build-mac` | macos-latest | .dmg |
| `build-android` | ubuntu-latest | .apk |
| `build-ios` | macos-latest | .ipa |

All jobs run in parallel. Artifacts are uploaded to the GitHub release.

### Manual Build Script

For local builds across all targets:

```bash
./scripts/build.sh [target]
```

**Targets:**

| Target | Description |
|--------|-------------|
| `linux` | Build Linux desktop (AppImage + deb) |
| `windows` | Build Windows desktop (.msi) |
| `mac` | Build macOS desktop (.dmg) |
| `android` | Build Android (.apk) |
| `ios` | Build iOS (.ipa) |
| `all-desktop` | Build all desktop targets |
| `all` | Build everything |

### Release Workflow

```bash
# 1. Update version
# Edit version in:
#   - desktop/src-tauri/tauri.conf.json
#   - mobile/capacitor.config.ts (if applicable)
#   - backend/pyproject.toml

# 2. Commit and tag
git add -A
git commit -m "chore: bump version to vX.Y.Z"
git tag vX.Y.Z
git push origin main --tags

# 3. CI/CD builds and publishes
# Artifacts appear in GitHub Releases
```

## 9. Data Storage

### File Structure

```
{DATA_DIR}/
├── dailybyte.db          # SQLite database
├── config.json            # App configuration (future)
└── media/
    ├── abc123.jpg         # Uploaded media files
    ├── def456.png
    └── ...
```

### Platform Defaults

| Platform | DATA_DIR |
|----------|----------|
| Linux (Tauri) | `~/.local/share/com.dailybyte.desktop/` |
| Linux (dev) | `~/.local/share/diarilinux/` |
| Windows | `%APPDATA%\dailybyte\` |
| macOS | `~/Library/Application Support/dailybyte/` |
| Dev mode | `./` (current directory) |
| Docker | `/data/` (volume-mounted) |

### Database

- **Engine:** SQLite with aiosqlite (async)
- **Location:** `{DATA_DIR}/dailybyte.db`
- **WAL mode:** Enabled automatically — allows concurrent reads during writes
- **Foreign keys:** Enforced (`PRAGMA foreign_keys=ON`) — cascading deletes active
- **Busy timeout:** 5 seconds — prevents immediate "database is locked" errors
- **Connection pool:** `pool_size=1` for SQLite (single writer), configurable for PostgreSQL
- **FTS:** Full-text search via FTS5 virtual table, auto-rebuilt on startup if stale
- **Migrations:** Auto-creates tables on startup (`Base.metadata.create_all`) + lightweight column migrations via `_COLUMN_MIGRATIONS`
- **Backup:** WAL checkpoint before export; atomic restore with rollback on failure

### Media Storage

- Uploaded files stored in `{DATA_DIR}/media/`
- File names are UUIDs to prevent collisions
- Original filename preserved in database metadata
- MIME type validated on upload (magic number check)
- Blocked file types: executables, scripts, HTML

## 10. Security Considerations

### Production Checklist

- [ ] Change `SECRET_KEY` from default value
- [ ] Set `APP_ENV=production` (disables `/docs`, restricts CORS)
- [ ] Configure `CORS_ORIGINS` to your actual domain
- [ ] Use HTTPS via reverse proxy (Nginx, Caddy)
- [ ] Set up rate limiting (built-in at 60/min, or proxy-level)
- [ ] Restrict file upload sizes (`MAX_MEDIA_SIZE_BYTES`)
- [ ] Regular database backups
- [ ] Run as non-root user (systemd, Docker)

### Built-in Security

| Feature | Implementation |
|---------|---------------|
| CORS | Configurable origins, credentials supported |
| Rate Limiting | In-memory per-IP, 60 req/min with memory leak protection |
| File Validation | Magic number check, MIME whitelist |
| Blocked Files | Executables, scripts, HTML blocked |
| Path Traversal | `..` sequences rejected in filenames |
| Archive Safety | Tar members validated before extraction (path traversal blocked) |
| Plugin Validation | entry_point restricted to safe dotted paths; stdlib modules blocked |
| Model Validation | Ollama model names validated before subprocess execution |
| XSS | DOMPurify sanitization on rendered HTML |
| Encryption | AES-256-GCM for entry-level encryption |
| Backup Atomicity | Restore backs up current DB first, rolls back on failure |
| Request Logging | UUID-based tracking with timing |
| Health Check | `/health` verifies database connectivity |
| Graceful Shutdown | Engine disposed, background enrichment tasks cancelled |

### Docker Security

```yaml
# Run as non-root user
services:
  dailybyte:
    user: "1000:1000"
    read_only: true
    tmpfs:
      - /tmp
    volumes:
      - dailybyte-data:/data
```

### Reverse Proxy Headers

When behind a reverse proxy, ensure these headers are forwarded:

```
X-Real-IP
X-Forwarded-For
X-Forwarded-Proto
```

This ensures correct rate limiting and URL generation.
